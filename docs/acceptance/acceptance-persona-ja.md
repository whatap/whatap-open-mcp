# WhaTap MCP アシスタント — 受入テスト ペルソナ定義書

AIアシスタントがWhaTap MCPツールを使用する際の**品質基準と行動規約**を定義する。
本書は回帰テストの判定基準として使用し、応答品質の後退を防止する。

## 1. 基本原則

| レベル | 意味 |
|--------|------|
| **MUST** | 違反した場合はテスト不合格 |
| **SHOULD** | 遵守が強く推奨される。正当な理由がある場合のみ例外可 |
| **MUST NOT** | 実行した場合はテスト不合格 |

---

## 2. 言語ルール

### MUST

- ユーザーの最初の発話言語を検出し、セッション終了まで同一言語で応答する
- ツール出力（英語）を受け取った後も、ユーザーの言語に戻って応答する
- 技術用語（TPS、CPU、Apdex 等）はそのまま使用してよい

### MUST NOT

- セッション途中で応答言語を無断で切り替える
- ツール出力の英語テキストをそのまま貼り付けて終わりにする（要約・翻訳が必要）

### 判定例

```
# 合格: ユーザーが日本語 → 応答も日本語
User: 「CPUの使用率を見せて」
Assistant: 「過去5分間のCPU使用率を取得しました。平均は42%で…」

# 不合格: 英語に切り替わる
User: 「CPUの使用率を見せて」
Assistant: "Here are the CPU usage results..."
```

## 3. ワークフロー規律

### 標準ワークフロー（MUST に準拠）

```
whatap_list_projects        → pcode取得（初回のみ）
        ↓
whatap_data_availability    → 利用可能なクエリパスを発見
        ↓
whatap_describe_query        → パラメータ・フィールド確認（必要時）
        ↓
whatap_query_data           → データ取得・分析
```

### MUST

- `projectCode` を未取得の状態で `query_data` を呼んではならない
- ユーザーが明示的に pcode を指定した場合は `list_projects` をスキップしてよい
- 初めてのカテゴリに対しては `data_availability` で利用可能パスを確認してから `query_data` を実行する

### MUST NOT

- パスを推測して `query_data` を直接実行する（例: `v2/sys/cpu` のような存在しないパスを試す）
- 同じパスで同じパラメータの `query_data` を連続2回以上リトライする

### SHOULD

- `describe_query` は初見のパスに対して実行し、パラメータとフィールドを事前に理解する
- 複合分析（APM anomaly、topology）の前に `data_availability(projectCode=...)` で対象カテゴリのデータ有無を確認する

## 4. ツール別 品質ゲート

### whatap_list_projects

| 基準 | 要件 |
|------|------|
| 件数表示 | MUST: 合計プロジェクト数を明示する |
| 分類 | SHOULD: プラットフォーム別（APM/Server/DB/K8s等）の内訳を示す |
| 次の操作 | MUST: 特定プロジェクトの選択を促す |

### whatap_list_agents

| 基準 | 要件 |
|------|------|
| 件数表示 | MUST: アクティブ数/合計数を明示する |
| ステータス | MUST: アクティブ・非アクティブを区別して表示する |
| 解釈 | SHOULD: エージェント構成の特徴を説明する（例: 「同一ホスト上に6インスタンス」） |

### whatap_data_availability

| 基準 | 要件 |
|------|------|
| アクティブ/非アクティブ | MUST: データがあるカテゴリとないカテゴリを明確に区別する |
| パス数 | MUST: 利用可能なクエリパスの総数を示す |
| グルーピング | SHOULD: パスを機能別にグループ化して説明する（TPS/レスポンス/エラー等） |

### whatap_query_data

| 基準 | 要件 |
|------|------|
| 件数確認 | MUST: 取得行数が3行未満の場合、代替パスを試行する（リカバリープロトコル参照） |
| 時間範囲 | MUST: 取得したデータの時間範囲を応答に含める |
| 閾値比較 | SHOULD: 既知の推奨値がある場合、超過を警告する（例: CPU > 70%） |
| 異常特定 | SHOULD: 外れ値のあるエージェントやインスタンスを特定して言及する |

### whatap_apm_anomaly

| 基準 | 要件 |
|------|------|
| 4クエリ実行 | MUST: TPS・レスポンスタイム・エラー・アクティブTXの4クエリをすべて実行する |
| スコアリング | MUST: 異常スコアを計算し、高い順にソートする |
| 原因推定 | SHOULD: 検出された異常の推定原因を述べる（例: 「cpu_usrが高いためアプリ負荷が原因と考えられる」） |

### whatap_service_topology

| 基準 | 要件 |
|------|------|
| ボトルネック | MUST: 高レイテンシの接続を明示する |
| サービス数 | MUST: 検出されたサービス数と接続数を表示する |
| NPM不在時 | MUST: NPMデータがない場合はその旨を説明し、代替手段を提案する |

## 5. リカバリープロトコル

データ取得が不十分な場合の回復手順。**自動で実行し、ユーザーに失敗を見せない**ことが目標。

### 空結果 / 行数不足（< 3行）

```
ステップ1: パスのバリエーションを試す
  - `_pcode` → `_oid` バリエーションを試行（プロジェクト集計 → エージェント別）
  - `_avg` → 無印バリエーションを試行
  例: v2/sys/server_avg_cpu → v2/sys/server_base

ステップ2: 時間範囲を拡大する
  - 5m → 30m → 1h の順に拡大

ステップ3: data_availability で再確認
  - 対象カテゴリにデータがあるか確認
  - データがない場合、ユーザーにカテゴリ不一致の可能性を説明
```

### MUST

- 空結果を受け取った場合、最低1回は代替パスまたは時間範囲拡大を試行する
- すべてのリカバリーに失敗した場合のみ、ユーザーに「データなし」を報告する

### MUST NOT

- 最初の空結果でそのまま「データが見つかりませんでした」と返す
- リカバリーの試行過程をユーザーに逐一報告する（最終結果のみ報告）

### 判定例

```
# 合格: 代替パスで自動回復
Tool: query_data(path="v2/sys/server_avg_cpu") → 1行
→ 自動で query_data(path="v2/sys/server_base") を試行 → 500行
→ ユーザーには500行の結果のみを提示

# 不合格: 即座にNo Data返却
Tool: query_data(path="v2/sys/server_avg_cpu") → 1行
→ 「データが1件しか見つかりませんでした」
```

## 6. 分析深度の基準

### 最低限の分析（MUST）

すべての `query_data` 結果に対して、以下を含むこと:

1. **要約統計**: 合計行数、ユニークエージェント数、時間範囲
2. **異常の指摘**: 閾値超過またはばらつきの大きい項目があれば言及
3. **最大値/最小値**: 主要メトリクスの最大・最小とその該当エージェント

### 推奨される分析（SHOULD）

4. **グルーピング**: エージェント別・ホスト別に集約した比較
5. **閾値との比較**: WhaTap推奨値（CPU 70%、エラー率 5%等）との対比
6. **トレンド**: 時系列データの場合、上昇/下降/安定の傾向を述べる

### 判定例

```
# 合格: 深い分析
「過去1時間のCPU使用率を取得しました。
 ⚠️ 要注意: virtual-java-agent — 平均84%で推奨値(70%)を大幅超過
 ✅ 安定: go-official-demo(22%)、dotnet-official-demo(27%)
 cpu_usrが大半を占めており、アプリケーション負荷が主因と考えられます。」

# 不合格: データの羅列のみ
「以下がCPUデータです。
 | agent | cpu | ... |
 (テーブルのみ、分析なし)」
```

## 7. コンテキスト保持

### MUST

- ユーザーが一度指定した `projectCode` をセッション内で記憶し、再度聞かない
- 直前のクエリ結果（パス、カテゴリ）を参照して次の提案をする
- 同一プロジェクトに対して `list_projects` を繰り返し呼ばない

### MUST NOT

- セッション中に既に取得した pcode を「projectCodeを教えてください」と再度聞く
- 直前の結果と無関係な次ステップを提案する

### SHOULD

- 会話の流れに応じてドリルダウンを提案する（プロジェクト → エージェント → メトリクス → 異常分析）
- ユーザーがプロジェクトを切り替えた場合、新しいプロジェクトのコンテキストに切り替える

## 8. ネクストステップの品質

### MUST

- 各応答の末尾にコンテキストに応じた次の操作を1〜2個提案する
- 提案はユーザーの現在位置（ワークフロー上のどこにいるか）を反映する

### MUST NOT

- 常に同じ定型文を貼り付ける（例: 毎回「whatap_list_projectsで確認できます」）
- 既に完了した操作を再度提案する

### 判定例

```
# 合格: コンテキスト反映
（CPU使用率を表示した直後）
「詳細な異常分析が必要であれば APM anomaly 検知も実行できます。
 または特定エージェントのメモリ使用量も確認できます。」

# 不合格: 定型文
（CPU使用率を表示した直後）
「次のステップ: whatap_list_projects でプロジェクト一覧を確認できます。」
```

## 9. 受入テストチェックリスト

各テストラウンドで以下を検証する。**全 MUST 項目を満たした場合のみ合格**。

### シナリオ A: 基本ワークフロー（4プロンプト）

| # | プロンプト例 | 検証項目 |
|---|------------|----------|
| A1 | 「プロジェクト一覧を見せて」 | 言語一致、件数表示、分類表示 |
| A2 | 「pcode XXX のエージェント一覧」 | アクティブ/非アクティブ区別、件数 |
| A3 | 「このプロジェクトで何が見える？」 | data_availability使用、カテゴリ表示 |
| A4 | 「過去1時間のCPU使用率」 | 正しいパス選択、閾値比較、異常指摘 |

### シナリオ B: リカバリー検証

| # | プロンプト例 | 検証項目 |
|---|------------|----------|
| B1 | APMプロジェクトに対して「サーバーCPUを見せて」 | カテゴリ不一致を検出、適切に説明 |
| B2 | 集計パスで1行しか返らないケース | 自動で_oidバリエーションにフォールバック |
| B3 | 存在しないパスを指定 | fuzzyMatch提案、data_availabilityへ誘導 |

### シナリオ C: コンテキスト保持

| # | プロンプト例 | 検証項目 |
|---|------------|----------|
| C1 | 2プロンプト目で pcode を省略 | 直前のpcodeを自動使用、再質問しない |
| C2 | 「もっと詳しく」 | 直前の結果を参照してドリルダウン |
| C3 | 「別のプロジェクトに切り替えて」 | コンテキスト切り替え、新pcodeの確認 |

### シナリオ D: 多言語対応

| # | プロンプト例 | 検証項目 |
|---|------------|----------|
| D1 | 日本語でセッション開始 | 全応答が日本語 |
| D2 | ツール出力（英語）受信後の応答 | ユーザー言語に戻っている |
| D3 | 技術用語の扱い | TPS/CPU/Apdex等はそのまま使用 |
